# Midterm Project: QA Implementation & Empirical Analysis

## 1. System Description

- Architecture
  - `frontend/` is a Next.js 15 + React 19 web application.
  - `backend/` is a Go Gin API backed by PostgreSQL, Redis, Swagger, and Prometheus.
  - QA automation lives at repo root with Playwright, reusable fixtures, mock backend helpers, generated logs, and evidence charts.
- Technologies
  - Playwright for E2E and API automation
  - Go testing for backend unit/integration coverage
  - Node.js scripts for metrics extraction, coverage summarization, and quality gates
  - GitHub Actions for CI/CD
  - Docker Compose for live backend API execution in CI
- Key functionalities
  - login and authorization
  - catalog listing and product details
  - cart management
  - checkout and payment-link initiation
  - order viewing, status transitions, and cancellation

## 2. Methodology

- Risk-based testing approach
  - Assignment 1 identified the top five high-risk modules: authentication and authorization, cart management, checkout and payment initiation, order lifecycle, and product catalog and listing.
  - Midterm re-used those five modules and re-scored them from actual automation evidence in `logs/module-risk-reevaluation.json` and `qa-docs/tables/risk-reevaluation.csv`.
- Test design strategy
  - Preserve the Assignment 2 smoke, critical, and regression structure.
  - Add backend unit/integration tests where coverage was stale or missing.
  - Add new Playwright cases only for high-risk modules and only for failure, edge, concurrency, and invalid-user behavior.
  - Separate runtime-dependent live API evidence from blocking mock-backed evidence.
- Automation tools used
  - Go `go test` with `-coverprofile` and `-json`
  - Playwright JSON reports for smoke, critical, regression, E2E, and API suites
  - Node scripts for failure extraction, flaky detection, coverage gaps, unexpected behavior, and gate evaluation

## 3. Automation Implementation

- CI/CD setup
  - `.github/workflows/qa-assignment2.yml` now installs dependencies, runs frontend lint, runs backend tests with coverage, runs smoke/critical/regression/E2E suites, starts the backend stack with Docker Compose, runs live API tests, generates metrics, enforces quality gates, uploads artifacts, and tears the stack down.
- Test structure
  - Backend tests
    - updated auth JWT tests
    - updated auth service tests
    - updated cart/checkout service tests
    - updated product service tests
    - new order service tests
  - Playwright midterm tests

    - `tests/e2e/checkout/checkout.midterm.spec.ts`

  - `QG-MID-002`: zero non-runtime critical failures
  - `QG-MID-003`: average high-risk backend coverage `>= 20%`
  - `QG-MID-004`: flaky rate `<= 5%`
  - `QG-MID-005`: non-runtime smoke pass rate `>= 85%`
  - Current evaluation from `logs/quality-gates-summary.json`: all five gates passed in this session.

## 5. Discussion

- What worked
  - The backend test layer is now green and produces real coverage artifacts.
  - Metrics, failure summaries, quality gates, and charts are generated from actual logs instead of placeholder values.
  - Runtime-dependent failures are now separated from blocking mock-backed evidence, which makes the quality gates more honest and more useful.
- What did not
  - Local sandbox execution could not complete browser-based Playwright runs for the new E2E tests because the frontend web server could not bind to `127.0.0.1:3000`.
  - High-risk backend coverage is still low for cart, orders, and catalog, so detectability is not yet where it should be.
- Unexpected findings
  - The checked-in `.env.qa` had live API toggles enabled, which created false-negative failures when backend services were unavailable.
  - Existing Assignment 2 backend tests had drifted from current interfaces and no longer represented trustworthy coverage until repaired.
- Next improvements
  - Add direct order-handler and catalog-service tests to lift the lowest module coverage areas.
  - Add frontend unit/component instrumentation if the stack is extended with a JS/TS unit-test runner.
  - Persist multi-run history so flaky detection is based on more than the current set of reports.
